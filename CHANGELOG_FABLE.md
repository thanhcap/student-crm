
## fable/email-automation-canvas — Parts 2–5 (canvas, graph engine, enrollment overlay, polish)

Builds on the foundation entry below (migration/backfill/rename were already live). No new dependency added — the canvas is hand-rolled with SVG bezier edges + absolutely-positioned node cards + pointer events (reactflow not needed; bundle stays lean).

### Part 2 — Visual canvas
- `AutomationCanvas` component (module-level in `src/app/page.js`): pannable (drag empty space), zoomable (wheel via non-passive native listener + −/％/＋/Fit controls), dotted-grid background; `canvas_zoom`/`canvas_offset` persist per sequence (debounced).
- Nodes = `sequence_steps` rows rendered as fixed-size cards at `(pos_x,pos_y)` with type icon, uppercase label, one-line config summary, colored left accent per type, green ✓ when fully configured / amber dot when incomplete, and 8px-grid-snapped dragging (position persists on drop, debounced).
- Edges = `sequence_edges` rows drawn as SVG beziers from bottom output ports to top input ports; condition nodes expose two labeled ports (Yes = green, No = gray → `branch='yes'|'no'`); one edge per output port (reconnecting replaces); click an edge → highlight + × to delete; Delete/Backspace also works; dashed preview while drawing; drop-target hit test on the target card's top strip.
- Left palette (drag onto canvas or click to add near viewport center): Email / Wait / Condition / LinkedIn view / LinkedIn connect / Call / Manual task / Goal. Trigger is auto-present, unique, undeletable.
- Right config panel slides in on node select; per-type fields write to `config` jsonb AND the legacy columns (`subject`/`body`/`subject_b`/`wait_days`/`condition`/`channel`/`task_note`) via `patchForNodeConfig` so the existing engine keeps working; merge-tag chips (same tokens as `resolveMergeTags`), template picker on email nodes; trigger panel maps to `email_sequences.trigger_type/trigger_value` (manual / new relationship / tag applied / **deal stage** — the deal-stage auto-enroll hook was added to `handleUpdateDealStage`).
- Header bar: back, inline-editable name, Active toggle (writes `is_active` AND mirrors legacy `status`), Saving…/Saved ✓ indicator (all canvas writes debounced through one `canvasPersist` helper), Duplicate (now copies node_type/pos/config and remaps edges), Delete.
- **Manual test:** open Email Automation → open a campaign → drag an Email node from the palette, connect trigger→email, drag it around, reload — position/zoom/pan restored; select node → edit subject → card summary updates and "Saved ✓" appears; DB row shows both `config.subject` and `subject` updated.

### Part 3 — Graph-aware execution engine (client + runner v4)
- ONE shared pure walker used everywhere: `resolveNextNode(enrollment, nodes, edges, sends)` → `{action: email|task|goal|wait|complete, node, waitDays}` plus `advanceAfterNode`. Client copy in page.js and `supabase/functions/sequence-runner/_shared/sequence-logic.ts` are kept behaviorally identical (comment markers on both).
- Semantics: trigger nodes pass through; an entry wait counts as elapsed (whoever set the pointer applied the delay to `next_send_at`); a wait encountered mid-walk pauses (`next_send_at = today + days`); condition nodes evaluate open/reply from the enrollment's `sequence_sends` rows (same tracking data as before) and follow yes/no; goal → enrollment `completed` (`stopped_reason='goal_reached'`); dead end / missing branch / cycle → `completed`. **Graceful degradation:** with no tracking data, `if_opened`/`if_replied` are false → the No branch is followed (documented in the condition panel; manual new-tab sends still can't be open-tracked — pre-existing gap, unchanged).
- Backward compat: nodes keep honoring the legacy `wait_days` column on action nodes (`arrivalDelayDays`) and legacy per-step `condition` skips (via existing `stepConditionMet`); sequences with no edge rows run through `syntheticChainEdges` (implicit straight chain); legacy enrollments with null `current_node_id` map `current_step` onto the ordered non-trigger chain. **This also fixes a latent off-by-one:** the backfilled trigger node (step_order −1) had shifted the old `steps[current_step]` indexing in runner v3 and the Outbox.
- `handleSendSequenceStep` (manual/Outbox path) and `enrollClientInSequence` (starts at the node after the trigger, `current_node_id` set) now walk the graph; Outbox (`dueSequenceSends`) resolves the actual due node (email or task) and labels the button accordingly; a one-shot housekeeping effect finalizes due enrollments that resolve to goal/dead-end/wait so nothing sits in limbo when auto-send is off.
- **Runner v4 deployed** (Supabase edge function `sequence-runner`, version 6): fetches `sequence_edges`, walks the same graph, creates tasks for LinkedIn/call/manual nodes, honors goal/wait/complete, and now **skips any sequence with `is_active` false** (the Active toggle is the runner gate). Gmail send path/tracking/quota/window logic unchanged.
- **Manual test (run + verified):** 16-scenario unit test of the extracted walker (bun): legacy chain order preserved incl. `wait_days=3` timing (A1–A6), replied→legacy-condition skip sends next step immediately (B1), canvas graph email→wait→condition yes/no→goal (C1–C4), missing branch completes gracefully (C5), mid-walk wait pauses (C6), cycle guard (C7), LinkedIn task action + dead-end completion (C8–C9). ALL PASSED. Live DB check: seq 1 = trigger(4)→1→2→3 edges, both active enrollments' `current_node_id=2` matches their `current_step=1` — same send order as before.

- Enrollments side pane: per-relationship current node (icon + summary), status badge, next-send date, Stop (reuses `handleStopEnrollment`), and Send now → when due.
